# Redis

# Q&A

Redis为什么快？全内存存储 + 高效数据结构 + 单线程无锁 + 内存优化 + 网络优化
- 内存读写

- 高效数据结构
| 数据类型       | 底层结构                | 优势                         |
| ---------- | ------------------- | -------------------------- |
| String     | dynamic string      | 直接连续内存存储，O(1) 读写           |
| Hash       | ziplist / hashtable | 小对象紧凑存储，大对象哈希表快速查找         |
| List       | quicklist (压缩链表+跳表) | 顺序访问快，范围操作高效               |
| Set        | intset / hashtable  | 小整数集合紧凑存储，大集合哈希表快速查找       |
| Sorted Set | skiplist + dict     | 范围查询和有序操作快速，插入、删除 O(log N) |

- 单线程+事件驱动
Redis 核心是单线程+事件循环（基于 epoll/kqueue）
没有多线程锁竞争问题 → 无锁开销
同步操作即可完成请求，避免上下文切换
对慢操作有阻塞风险，但在内存小、操作快的情况下非常高效

- 高效内存管理
Redis 自带内存分配优化（jemalloc 或 tcmalloc）
内存碎片小，分配释放开销低
数据压缩策略（ziplist、intset）减少内存占用，提高 CPU cache 命中

- 其他
批量操作：MSET / MGET 可以一次性处理多条请求，减少网络往返
管道（Pipeline）：多条命令打包发送 → 减少网络延迟
持久化异步：RDB / AOF 都可以异步写磁盘，不阻塞主线程
