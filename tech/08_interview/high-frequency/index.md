# 高并发&高可用

## 什么是高并发、高可用

高并发：以公路比喻系统，那就是 更宽、更短
    更宽：不改系统性能，网络带宽、服务器数量
    更短：缓存 直接提供更短的路；优化 让长度变短（分层缓存_eg:mybatis_cache，逻辑串改并，
高可用：健壮 避免单点突破，同时保证全局


## 如何排查性能问题

方法论：监控 & 复现 & 定位

1. 定位接口
    明确性能瓶颈在哪个接口或模块
    关注响应时间、吞吐量、错误率
2. 定位代码
    使用实时堆栈、调用链或 Profiler
    找到耗时热点、频繁调用点
    分析内存占用、锁竞争、IO 阻塞
3. 优化
    算法优化
        优化时间复杂度，减少重复计算
        串行改并发，提升 CPU 利用率
        IO 改缓存/预处理，降低磁盘或网络瓶颈
    链路优化
        接口拆分，降低单次请求复杂度
        剔除非必要逻辑，减少计算或数据处理量
    系统优化
        增加资源（CPU、内存、IO 带宽）
        减少单机负载，平衡负载压力
    架构优化
        启用性能更好的中间件或方案
        引入缓存、消息队列、异步处理
        数据库或服务拆分，支持水平扩展
4. 验证与迭代
    复测性能指标，确认优化效果
    保证功能正确性与稳定性
    持续监控，形成优化闭环

## 如何优化性能

方法论：算法 & 代码链路 & 系统资源 & 架构

1. 算法与数据结构层面
    优化时间复杂度、空间复杂度
    选择合适的数据结构（如哈希表 vs 数组）
    避免重复计算（缓存、记忆化）
2. 代码实现层面
    减少不必要的循环或调用
    使用高效库函数（如 Java Stream vs for 循环要权衡）
    并发/异步处理
3.  系统资源层面
    内存管理（对象复用、垃圾回收调优）
    CPU利用率优化（多线程、分布式计算）
    IO优化（批量处理、延迟加载）
4. 架构层面
    缓存（Redis / 本地缓存）
    数据库索引优化、SQL优化
    分布式架构与负载均衡
    异步队列处理高峰请求

## 缓存不一致怎么解决

| 方法                          | 描述                   | 优缺点                 |
| --------------------------- | -------------------- | ------------------- |
| **Cache-Aside + 删除缓存**      | 数据库更新后删除缓存，下次读取再刷新   | 简单可靠，但读写之间可能有短暂不一致  |
| **写缓存+写数据库（Write-Through）** | 写操作同步更新缓存和数据库        | 强一致性最好，但写延迟高        |
| **分布式锁**                    | 对同一数据加锁，保证顺序更新       | 可解决并发冲突，但增加复杂度和性能开销 |
| **消息队列 + 异步刷新**             | 数据库写后通过 MQ 同步更新各节点缓存 | 分布式可用，但异步存在短暂不一致    |
| **版本号 / 时间戳**               | 缓存记录数据版本号，更新时比较版本    | 可以防止旧数据覆盖新数据，适合并发场景 |
